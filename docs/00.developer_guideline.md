## 📌 1. 프로젝트 개요

본 프로젝트는 사용자가 상품을 탐색하고, 장바구니에 담아 결제까지 진행할 수 있는  
온라인 쇼핑몰 웹 애플리케이션을 구현하는 것을 목표로 합니다.

### 🎯 목적
사용자가 상품을 쉽게 탐색하고 구매할 수 있도록  
상품 목록, 상세 페이지, 장바구니, 결제 기능을 포함한 전체 쇼핑 흐름을 구현하는 데 목적이 있습니다.

### 📦 범위
본 프로젝트는 다음 기능을 포함합니다.
- 상품 목록 조회
- 상품 상세 정보 확인
- 장바구니 담기 및 수량 관리
- 결제 기능 및 결제 완료 페이지
- 페이지 간 라우팅 및 데이터 흐름 관리

### 🧩 배경
온라인 쇼핑 서비스는 단순한 기능 구현을 넘어  
사용자의 편리한 경험과 자연스러운 흐름이 중요합니다.  

이에 따라 본 프로젝트는  
상품 탐색부터 결제 완료까지 이어지는 사용자 중심의 흐름을 설계하고 구현하는 데 초점을 두었습니다.

### 💡 제공 가치
본 프로젝트는 다음과 같은 가치를 제공합니다.
- 직관적인 사용자 경험 기반의 쇼핑 흐름 제공
- 컴포넌트 기반 구조를 통한 유지보수 용이성 확보
- Storybook을 통한 UI 문서화
- Cypress를 통한 사용자 흐름 기반 테스트 환경 구축

## 🏗️ 2. 시스템 아키텍처

본 프로젝트는 React 기반의 단일 페이지 애플리케이션(SPA) 구조로 구현되었으며,  
상품 목록부터 상세 페이지, 장바구니, 결제 완료 페이지까지 하나의 자연스러운 사용자 흐름으로 연결되도록 설계하였습니다.

---

### 2-1. 전체 구조

```text
사용자
  ↓
상품 목록 페이지 (ProductListPage)
  ↓
상품 상세 페이지 (ProductDetailPage)
  ↓
장바구니 페이지 (CartPage)
  ↓
결제 모달 (PaymentModal)
  ↓
결제 완료 페이지 (PaymentCompletePage)

###2-2. 컴포넌트 구조
App
 ├── Router
 │    ├── ProductListPage
 │    ├── ProductDetailPage
 │    ├── CartPage
 │    └── PaymentCompletePage
 │
 ├── CartList
 │    └── CartItem
 │
 ├── PaymentModal
 │
 └── 공통 컴포넌트
      └── Button

###2-3. 데이터 흐름
App (상위 상태 관리)
 └── cartItems 상태 보유
      ↓
 ProductListPage
 ProductDetailPage
 CartPage
      ↓
 PaymentModal
      ↓
 PaymentCompletePage
-장바구니 데이터는 App에서 관리합니다.
-각 페이지는 props를 통해 동일한 장바구니 데이터를 전달받습니다.
-사용자의 클릭 이벤트(장바구니 담기, 수량 변경, 삭제, 결제 완료)는 하위 컴포넌트에서 발생하고, 처리 로직은 상위에서 관리합니다.

즉,
데이터는 상위에서 하위로 전달되고, 이벤트는 하위에서 상위로 전달되는 구조입니다.

2-4. 각 컴포넌트 역할
App
-전체 페이지를 연결하는 최상위 컴포넌트
-라우팅 설정
-장바구니 상태 관리
ProductListPage
-상품 목록을 보여주는 페이지
-상품 클릭 시 상세 페이지로 이동
-장바구니 담기 상태를 표시
ProductDetailPage
-상품 상세 정보를 보여주는 페이지
-수량 선택 및 장바구니 담기 기능 제공
-관련 상품 표시
CartPage
-장바구니에 담긴 상품을 관리하는 페이지
-수량 변경, 삭제, 금액 계산 기능 제공
-결제하기 버튼 제공
CartList / CartItem
-장바구니 상품 목록과 개별 상품 정보를 표시하는 컴포넌트
-재사용 가능한 구조로 분리
PaymentModal
-결제 직전 정보를 보여주는 모달
-최종 결제 금액 확인 및 결제 완료 처리
PaymentCompletePage
-주문 완료 결과를 보여주는 페이지
-사용자에게 결제 성공 여부와 안도감을 제공

2-5. 연동 방식

각 페이지는 단순히 연결만 되는 것이 아니라,
사용자가 이전 페이지에서 수행한 동작이 다음 페이지에도 반영되도록 설계되었습니다.

예를 들어,

-상세 페이지에서 상품을 장바구니에 담으면
-장바구니 페이지에서 해당 상품이 바로 보이고
-다시 상품 목록으로 돌아가도 담긴 상태가 유지됩니다.

이와 같이 본 프로젝트는 페이지 간 이동과 데이터 일관성을 함께 고려한 구조입니다.

2-6. 관련 문서

## 🧭 4. 개발 규칙 및 주요 결정 사항

프로젝트를 안정적으로 진행하고, 이후 유지보수와 협업이 가능하도록  
브랜치 전략, 코딩 스타일, 커밋 메시지 규칙을 아래와 같이 정의합니다.

---

### 4-1. Git 브랜치 전략

본 프로젝트는 개인 프로젝트 형태로 진행되었지만,  
협업 확장을 고려하여 다음과 같은 브랜치 전략을 기준으로 삼습니다.

#### 기본 브랜치
- `main` : 최종적으로 안정된 코드만 반영하는 브랜치

#### 작업 브랜치 예시
- `feature/product-detail-page`
- `feature/cart-flow`
- `refactor/cart-state`
- `docs/shopping-requirements`

#### 운영 방식
1. 새로운 기능은 별도 브랜치에서 작업
2. 기능 단위로 테스트 및 정리 후 `main`에 반영
3. 문서 수정도 기능과 분리하여 관리

#### 기대 효과
- 기능별 작업 이력을 명확하게 관리할 수 있음
- 코드 충돌을 줄일 수 있음
- 어떤 작업을 했는지 추적이 쉬움

---

### 4-2. 코딩 스타일 가이드

프로젝트 전반의 일관성을 유지하기 위해 아래 기준을 따릅니다.

#### 기본 원칙
- 컴포넌트는 한 가지 역할에 집중하도록 작성
- 중복 코드는 줄이고 재사용 가능한 구조를 우선 고려
- 상태는 가능한 상위 컴포넌트에서 관리
- 변수명과 함수명은 의미가 드러나도록 작성

#### 네이밍 규칙
- 컴포넌트: PascalCase  
  예: `CartItem`, `ProductDetailPage`
- 함수 / 변수: camelCase  
  예: `handleAddToCart`, `paymentSummary`
- 파일명: 컴포넌트명과 동일하게 작성  
  예: `CartList.js`, `PaymentModal.js`

#### 코드 작성 기준
- 조건문과 반복문은 읽기 쉽게 작성
- 이벤트 처리 함수는 `handle` 접두어 사용
- 복잡한 로직은 작은 함수로 분리
- UI와 상태 로직이 과하게 섞이지 않도록 구성

#### 스타일링 기준
- 현재는 Inline CSS를 사용하지만, 일관된 간격과 배치를 유지
- 버튼, 카드, 영역 구분이 명확하게 보이도록 작성
- 사용자 행동에 따른 상태(활성/비활성)가 시각적으로 드러나게 구성

---

### 4-3. 커밋 메시지 규칙

커밋 메시지는 작업 내용을 한눈에 파악할 수 있도록  
의미 단위로 작성합니다.

#### 타입 규칙
- `feat` : 기능 추가
- `refactor` : 구조 개선 및 리팩터링
- `docs` : 문서 작성 및 수정
- `fix` : 오류 수정
- `chore` : 기타 정리 작업

#### 작성 예시
```bash
git commit -m "feat: add product detail page"
git commit -m "refactor: lift cart state to app"
git commit -m "docs: update shopping requirements"
git commit -m "fix: correct payment modal navigation"

작성 원칙
-한 커밋에는 하나의 주제만 담기
-무엇을 했는지가 바로 드러나게 작성
-모호한 표현보다 기능 중심 표현 사용

4-4. 프로젝트 진행 중 중요했던 결정

프로젝트를 진행하면서 몇 가지 중요한 결정을 내렸고,
이 결정들은 전체 방향과 구조에 큰 영향을 주었습니다.

1) 상태를 상위 컴포넌트에서 관리하기로 한 결정

초기에는 각 컴포넌트 내부에서 상태를 관리했지만,
장바구니와 결제 흐름이 확장되면서 데이터가 여러 페이지에서 동시에 필요해졌습니다.

따라서 장바구니 상태를 상위 컴포넌트(App)에서 관리하는 구조로 변경하였습니다.

영향

-페이지 간 데이터 일관성 확보
-장바구니 상태 유지가 쉬워짐
-상세 페이지, 장바구니, 결제 페이지 연결이 자연스러워짐

2) 기능 구현 후 리팩터링을 반복하는 방식으로 진행한 결정

처음부터 완벽한 구조를 만들기보다,
기능을 먼저 구현한 뒤 점진적으로 구조를 개선하는 방식을 선택했습니다.

영향

-빠르게 동작하는 화면을 먼저 확인할 수 있었음
-구현 후 필요에 따라 컴포넌트를 분리할 수 있었음
-실무와 비슷한 개발 흐름을 경험할 수 있었음

3) Storybook을 이용해 컴포넌트를 문서화하기로 한 결정

버튼, 장바구니 아이템, 결제 관련 UI를 별도로 확인하고 설명할 수 있도록
Storybook을 도입했습니다.

영향

영향

-고객에게는 기능 중심으로 쉽게 설명 가능
-개발자에게는 구조와 구현 방식을 명확히 전달 가능
-문서 활용 목적이 분명해짐

5) 사용자 흐름을 중심으로 페이지를 연결하기로 한 결정

단순히 페이지를 추가하는 것이 아니라,
상품 목록 → 상세 페이지 → 장바구니 → 결제 → 완료의 흐름을 하나의 서비스 경험으로 연결하는 방향을 선택했습니다.

영향

-기능 단위가 아닌 서비스 단위로 설계하게 됨
-페이지 간 연동을 더 자연스럽게 만들 수 있었음
-사용자 경험을 우선하는 구조로 발전함

4-5. 정리

브랜치 전략, 코딩 스타일, 커밋 메시지 규칙은
프로젝트를 일관되게 유지하기 위한 기준입니다.

또한 프로젝트 진행 중 내린 주요 결정들은
데이터 일관성, 사용자 경험, 유지보수성, 문서화 방향에 직접적인 영향을 주었으며,
결과적으로 프로젝트를 더 구조적이고 확장 가능한 형태로 발전시키는 데 도움이 되었습니다.
